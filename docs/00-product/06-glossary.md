# 06 — Glossário

Termos canônicos do GlobalTracker. Quando dois termos parecem sinônimos, este glossário decide qual é canônico — o outro é "não confundir com".

| Termo | Definição canônica | Não confundir com |
|---|---|---|
| **Workspace** | Tenant lógico do GlobalTracker. Cada cliente/operador tem um. Todas as tabelas têm `workspace_id`. RLS no Postgres garante isolamento. | "Cliente" (genérico), "Conta" (Meta/Google têm seu próprio conceito). |
| **Lançamento (Launch)** | Janela de operação de marketing com tracking config próprio (Pixel ID, conversion actions, audiences). Tem `public_id` legível e UUID interno. | "Campaign" (entidade Meta/Google que vive **dentro** de um Lançamento). |
| **Página (Page)** | LP/checkout/sales registrado no GlobalTracker, com `event_config`, `allowed_domains` e `page_token`. Modos: `a_system`, `b_snippet`, `c_webhook`. | URL bruta — `pages.url` é apenas referência informativa. |
| **PageToken** | Identificador público escopado a uma página, embutido em snippet HTML. Hash em DB; rotação com janela de overlap (ADR-023). | "API key" — page_token não dá acesso a operações administrativas. |
| **Lead** | Indivíduo identificado por ao menos um identificador (email, phone, external_id) em um workspace. Tem PII hash + enc + `lead_public_id`. | "Visitor" — visitor é anônimo (cobra `__fvid`); lead tem identidade. |
| **Visitor (`__fvid`)** | Cookie anônimo first-party com UUID v4 gerado pelo tracker. Existe na Fase 3 para multi-touch. Em Fases 1–2, coluna `events.visitor_id` fica reservada (nullable). | Lead — visitor é pré-identificação. |
| **Lead Token (`__ftk`)** | Cookie first-party assinado (HMAC-SHA256) emitido por `/v1/lead`. Permite reidentificação em retornos sem reenviar PII. TTL 30–90d. Binding ao `page_token_hash`. | `lead_id` em claro — `__ftk` é claim opaco. |
| **Lead Alias** | Ligação `(workspace_id, identifier_type, identifier_hash) → lead_id` em `lead_aliases`. Substitui unique constraints em `leads`. | Lead — um lead pode ter múltiplos aliases. |
| **Lead Merge** | Fusão de N leads em um canonical (mais antigo). Auditado em `lead_merges`. Atualiza FKs em events/lead_attribution/lead_stages. | "Deduplicação" genérica — merge é canônico e auditado. |
| **Consent** | Registro em `lead_consents` por finalidade (analytics, marketing, ad_user_data, ad_personalization, customer_match). Snapshot por evento em `events.consent_snapshot`. | "Opt-in" — opt-in é mecanismo; consent é o registro auditável. |
| **Event** | Tupla `(workspace_id, event_id, event_name, event_time, ...)` em `events`. Imutável. Vem de tracker (PageView/Lead/Purchase) ou webhook normalizado. | "Hit" GA4 — semelhante mas GA4 tem campos próprios; event é interno. |
| **Raw Event** | Payload bruto recebido pelo Edge antes de normalização. Persistido em `raw_events` para o modelo "fast accept" (ADR-004). Retenção 7 dias. | Event — raw_event é pré-processamento; event é normalizado. |
| **Lead Stage** | Marcação em `lead_stages` indicando posição do lead no funil (`registered`, `purchased`, `watched_class_3`). Pode ser único ou recorrente. | Funnel completo — stage é um ponto; funnel é a sequência. |
| **Attribution** | Conjunto de campos (utm_*, click IDs, account_id/campaign_id/ad_id) que ligam um evento à origem de tráfego. | "Source" / "Medium" GA4 — attribution é mais granular (8+ campos). |
| **First-touch / Last-touch** | Modelos de atribuição. First-touch: origem inicial do lead em `(lead_id, launch_id)`. Last-touch: origem da conversão de lead. ADR-015. | "Multi-touch" — Fase 3+. |
| **Link curto / Redirector** | Slug em `links` que redireciona via `/r/:slug` para destino com UTMs propagados. Registra clique em `link_clicks`. | Bitly/short.io — sistema próprio, integrado ao tracker. |
| **Dispatch** | Envio de evento normalizado a destino externo (Meta CAPI, GA4 MP, Google Ads conversion, audience). Cada envio é um `dispatch_job`. | "Webhook out" — dispatch tem semantics próprios (eligibility, retry, idempotency_key, DLQ). |
| **Dispatch Job** | Linha em `dispatch_jobs` com status (`pending` → `processing` → `succeeded`/`failed`/`skipped`/`dead_letter`/`retrying`). Idempotency key derivada por destino (ADR-013). | "Queue message" — message é transporte; job é a unidade rastreável. |
| **Dispatch Attempt** | Cada tentativa de envio é registrada em `dispatch_attempts` com payloads sanitizados, response status, erro. | Job — um job tem múltiplos attempts. |
| **Audience** | Definição declarativa de público (em `audiences.query_definition`). Materializada em `audience_snapshots` + `audience_snapshot_members`. Strategy condicional para Google. | "Lista" — termo genérico; audience é entidade do GlobalTracker. |
| **Audience Snapshot** | Estado materializado de uma audience em um momento (`snapshot_id`, `snapshot_hash`, members). Permite cálculo de diff entre T-1 e T. | "Audience" — audience é definição; snapshot é estado. |
| **Audience Sync Job** | Job que envia diff entre snapshot anterior e atual para Meta/Google. Lock por `audience_id + platform_resource_id`. | Dispatch job — sync job opera em batch de leads, não em evento individual. |
| **Cost ingestor** | Cron diário que busca gasto Meta/Google e grava em `ad_spend_daily` com normalização cambial. | "Billing" — cost é gasto em mídia, não billing do GlobalTracker. |
| **FX (Foreign Exchange)** | Taxa cambial usada para normalizar `spend_cents` → `spend_cents_normalized` na moeda do workspace. Provedor configurável (OQ-001). | "Câmbio comercial" — FX aqui é técnico, não financeiro humano. |
| **Pixel Policy** | Atributo de `pages.event_config` definindo coordenação browser↔server: `server_only` / `browser_and_server_managed` / `coexist_with_existing_pixel`. ADR-011. | "Server-side tracking" — pixel policy é decisão por página, não global. |
| **Customer Match strategy** | Atributo de `audiences.destination_strategy` para Google: `google_data_manager` / `google_ads_api_allowlisted` / `disabled_not_eligible`. ADR-012. | "Audience type" Google — strategy é nossa decisão; type é da plataforma. |
| **Tracker.js** | Script TS vanilla < 15 KB gzipped instalado em LPs. Lê config, captura attribution, envia eventos, gerencia cookies `__fvid`/`__ftk`. | "Pixel" Meta — pixel é blob fechado; tracker.js é nosso. |
| **Edge Gateway** | Cloudflare Worker com Hono que recebe `/v1/*`. Faz validação síncrona + persistência em `raw_events` + 202. Não chama Meta/Google diretamente. | "Backend" — termo genérico; edge gateway é específico. |
| **Ingestion Processor** | Worker async (CF Queue consumer) que normaliza `raw_events` → `events`/`leads`/`lead_stages` e cria `dispatch_jobs`. | Edge Gateway — gateway é síncrono; processor é assíncrono. |
| **Dispatcher** | Worker async que consome `dispatch_jobs` e envia para destinos externos. Um dispatcher por destino (Meta CAPI, GA4, Google Ads, Audience). | Webhook adapter — adapter é entrada (in); dispatcher é saída (out). |
| **Webhook Adapter** | Handler de webhook **inbound** (Hotmart, Stripe, Kiwify) em `apps/edge/src/routes/webhooks/<provider>.ts`. Valida signature + normaliza payload. | Dispatcher — adapter é in; dispatcher é out. |
| **Idempotency Key** | Chave que garante que envio repetido não causa duplicata. Derivada como `sha256(workspace_id|event_id|destination|destination_resource_id|destination_subresource)` (ADR-013). | Event ID — event_id identifica o evento; idempotency key identifica o envio. |
| **PII Key Version (`pii_key_version`)** | Versão da chave AES-GCM usada para criptografar PII em um registro. Permite rotação sem downtime via lazy re-encryption (ADR-009). | "Encryption version" genérico — `pii_key_version` é específico do GlobalTracker. |
| **HKDF** | Função de derivação de chave (`workspace_key = HKDF(PII_MASTER_KEY_V{n}, salt=workspace_id, info="pii")`). | "PBKDF2" — outro algoritmo; HKDF é o usado aqui. |
| **SAR (Subject Access Request)** | Pedido legal (LGPD/GDPR) de erasure ou acesso. No GlobalTracker, executado via `DELETE /v1/admin/leads/:lead_id`. ADR-014. | "Data export" — export pode existir mas SAR aqui é erasure. |
| **Audit Log** | Tabela append-only `audit_log` registrando mutações em entidades de configuração + acessos a PII em claro. Retenção 7 anos. | "Activity log" UI — audit log é técnico/legal. |
| **DLQ (Dead Letter Queue)** | Destino final de mensagens/jobs que falharam após `max_attempts`. Permite reprocessamento manual. | "Failed jobs" — DLQ é fila; failed jobs é status. |
| **Replay protection** | Mecanismo que rejeita request com `event_id` já visto nos últimos 7 dias (cache em CF KV). ADR-021. | "Idempotency" — idempotency é correção de duplicata pós-aceito; replay protection rejeita pré-aceito. |
| **Event Time Clamp** | Regra do Edge que reescreve `event_time` para `received_at` quando offset > `EVENT_TIME_CLAMP_WINDOW_SEC` (default 300s). ADR-020. | "Server time" — clamp é específico para casos de offset grande. |
| **Page Token Rotation** | Substituição de `page_token` com janela de overlap (default 14 dias) durante a qual ambos são aceitos. ADR-023. | "Token refresh" OAuth — rotation é diferente. |
| **`event_id` interno** | Identificador único de evento dentro do workspace. Para webhook, derivado de `sha256(platform || ':' || platform_event_id)[:32]` (ADR-019). Para tracker, gerado client-side (UUID-like). | `event_id` Meta CAPI — Meta espera nosso event_id como `eventID`/`event_id` deles; mesma string usada para dedup. |
| **`lead_public_id`** | Identificador público do lead (UUID v4) usado em propagação cross-domain (LP→checkout). Não-PII, não-guessable. | `lead_id` interno (UUID em FKs). |
| **`fbc` / `fbp`** | Cookies Meta first-party para click ID (`fbc`) e browser ID (`fbp`). Capturados pelo tracker, propagados a CAPI. | `fbclid` — query param; `fbc`/`fbp` são cookies. |
| **`gclid` / `gbraid` / `wbraid`** | Click IDs Google em URL. `gbraid` para iOS app campaigns; `wbraid` para web app. | "Google click" genérico — três tipos distintos. |
| **`_gcl_au` / `_ga`** | Cookies Google first-party. `_gcl_au` é Google Ads conversion linker; `_ga` é GA4 client_id. Capturados, não criados. | `gclid` — cookie persiste; `gclid` é one-shot URL. |
| **Enhanced Conversions** | Recurso Google Ads que enriquece conversão original com PII hashada via Google Ads API adjustment. Exige `order_id` + tag original com click ID. | "Conversion upload" — enhanced conversions é evolução; upload é base. |
| **Data Manager API** | API Google que substitui Google Ads API para Customer Match em novos adotantes a partir de abril/2026. ADR-012. | "Google Ads API" — Data Manager é o novo path. |
| **Doc-sync** | Política de sincronização doc↔código no mesmo commit. Quando impossível, marcar `[SYNC-PENDING]` em `MEMORY.md §2`. | "Code review" — doc-sync é fonte de verdade alinhada. |
| **T-ID** | Identificador de tarefa atômica de execução em sprint (`T-<N>-<NUM>`). Cabe em UM PR. Tem ownership concreto + DoD. | "Issue GitHub" — T-ID é mais granular e estruturado. |
| **Onda de paralelização** | Grupo de 3–5 T-IDs `parallel-safe=yes` com ownership disjunto, executadas em paralelo por subagents. | "Sprint" — sprint contém múltiplas ondas. |
| **Pixel ID** | Identificador único de um Meta Pixel (15-16 dígitos). Configurado em `launches.config.tracking.meta.pixel_id`. Encontrável em `business.facebook.com → Events Manager → Pixel`. | Pixel browser script — ID é o número; pixel é o conjunto. |
| **CAPI (Conversions API)** | API server-side do Meta para envio de eventos de conversão. Substitui ou complementa o Pixel browser. Permite enriquecimento de `user_data` server-side (ADR-006). | Pixel — Pixel é client-side; CAPI é server-side. |
| **CAPI Token** | Token de acesso (Bearer) emitido pelo Meta para autenticar chamadas à Conversions API. Vinculado a App + Pixel. | Page token — page token é nosso, CAPI token é do Meta. |
| **AEM (Aggregated Event Measurement)** | Mecanismo do Meta para iOS 14.5+ que limita medição de eventos. Exige priorização dos 8 eventos mais importantes por domínio em `business.facebook.com → Events Manager → Aggregated Event Measurement`. | "iOS 14 limitation" — AEM é a solução; iOS 14 é o motivo. |
| **Domain Verification (Meta)** | Processo no Meta Business Manager para comprovar posse de um domínio. Pré-requisito para AEM e para evitar penalização do Pixel. Configurado em `business.facebook.com/settings/owned-domains`. | DNS — Domain Verification pode usar DNS, meta tag ou file upload. |
| **Test Event Code** | String configurável no Meta CAPI que marca evento como teste. Faz aparecer em Events Manager → Test Events sem afetar produção. Configurável em `META_CAPI_TEST_EVENT_CODE` ou por workspace. | Debug mode — Meta usa test event code; GA4 usa `debug_mode`. |
| **Match Quality Score (Meta)** | Score 0-10 retornado pelo Meta após dispatch indicando quão bem `user_data` permitiu match com usuário Facebook. Quanto mais campos enviados (em, ph, fbc, fbp, etc.), maior. | "Conversion accuracy" genérico — Match Quality é métrica específica do Meta. |
| **Measurement ID (GA4)** | Identificador de Data Stream do GA4 no formato `G-XXXXXXXX`. Configurado em `launches.config.tracking.google.ga4_measurement_id`. Encontrável em GA4 → Admin → Data Streams. | Property ID — Measurement ID identifica stream; Property ID identifica property. |
| **API Secret (GA4)** | Secret de autenticação do GA4 Measurement Protocol. Gerado em GA4 → Admin → Data Streams → API Secrets. | API key — termo genérico; API Secret é específico do MP. |
| **GA4 Measurement Protocol (MP)** | API server-side do GA4 para envio de eventos. Equivalente conceitual ao CAPI do Meta. Usa `client_id` para deduplicação com gtag browser. | gtag.js — gtag é client-side; MP é server-side. |
| **GA4 DebugView** | Ferramenta do GA4 que mostra eventos em tempo real para debug. Ativada via `debug_mode=1` no payload ou query param `&debug_mode=1`. Acessível em `analytics.google.com → Admin → DebugView`. | Realtime — DebugView é apenas para debug; Realtime é dashboard de produção. |
| **Conversion Action (Google Ads)** | Definição em Google Ads de qual conversão rastrear (compra, lead, signup) com janela de atribuição e modelo. Tem ID próprio. Mapeado em `launches.config.tracking.google.conversion_actions`. | Conversion event GA4 — Conversion Action é Google Ads, não GA4. |
| **Auto-tagging (Google Ads)** | Recurso do Google Ads que adiciona `gclid` automaticamente nas URLs de destino dos anúncios. Sem ele, `gclid` precisa ser configurado manualmente. Ativável em `ads.google.com → Configuração → Conta → Auto-tagging`. | Manual tagging — auto-tagging usa `gclid`; manual tagging usa UTMs custom. |
| **Test Mode (workspace)** | Estado temporário do workspace (TTL 1h) que faz tracker.js, Edge e dispatchers tratarem eventos como teste. Não afetam dashboards nem audiences. Toggle em `/launches/:id/events/live`. | Sandbox env — test mode é por workspace em produção; sandbox é env separado. |
| **Health Badge** | Componente visual ([07-component-health-badges.md](../70-ux/07-component-health-badges.md)) que mostra estado verde/amarelo/vermelho/cinza de uma seção ou recurso. Agregado de métricas underlying. | Status indicator genérico — Health Badge é o componente padronizado. |
| **Onboarding wizard** | Fluxo guiado de 5 passos ([03-screen-onboarding-wizard.md](../70-ux/03-screen-onboarding-wizard.md)) para configuração mínima funcional do workspace. Skippable; estado em `workspaces.onboarding_state`. | Tutorial — wizard tem ações reais; tutorial é didático. |
| **Design tokens** | Variáveis nomeadas (cor, tipografia, espaço, radius, shadow, motion) que toda UI consome via CSS variables. Definição canônica em [01-design-system-tokens.md](../70-ux/01-design-system-tokens.md); input auto-extraído em [`/DESIGN.md`](../../DESIGN.md). | Hex/px raw — proibidos em componentes; sempre token. |
| **WCAG 2.2 AA** | Standard de acessibilidade alvo do Control Plane. Inclui critérios novos do 2.2: target size 24×24px (2.5.8), focus appearance (2.4.11), dragging movements (2.5.7), consistent help (3.2.6). Detalhe em [10-accessibility.md](../70-ux/10-accessibility.md). | WCAG 2.1 — versão anterior, menos critérios. |
| **shadcn/ui** | Biblioteca de componentes React baseada em Radix Primitives + Tailwind. Não é dependência runtime — código copiado para `apps/control-plane/src/components/ui/` e customizado via tokens locais. | Material UI / Chakra — diferentes; shadcn é "owned code". |
| **Skill (Claude Code)** | Pacote em `.claude/skills/<name>/SKILL.md` com convenções/prompts reutilizáveis. Ex.: `design-system` que extrai tokens de URL. | Subagent — skill é prompt; subagent é tipo de agent (`.claude/agents/`). |
